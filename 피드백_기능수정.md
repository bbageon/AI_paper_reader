# 피드백 & 기능 수정 이력

각 항목 = 문제 사항 / 해결 전략 / 해결법 (각 섹션 5줄 이하).

---

## 1. 라이브러리 구조 — 단일 세션 → 파일시스템 기반

**문제 사항**
- PoC가 Flask 세션 하나당 PDF 하나만 보던 구조라 여러 논문 누적 불가.
- 폴더 분류·이동·삭제·이름 변경 같은 일반적 파일 관리 동작이 전부 없음.

**해결 전략**
- 디스크 트리(`data/library/<폴더>/<paper_id>/...`)를 라이브러리의 진실원천으로.
- DB 없이 JSON 파일들만으로 관리, UI 익스플로러가 트리를 그대로 노출.

**해결법**
- 폴더 = 실제 디렉토리, paper = 디렉토리 안 고정 파일들(`original.pdf` / `content.md` / `notes.md` / `meta.json` / `chat.json`).
- `/api/library/*` 와 `/api/papers/<id>/*` 엔드포인트로 CRUD 노출, 컨테이너 down 해도 `./data` 볼륨에 유지.

## 2. PDF 뷰어가 1페이지만 보이고 텍스트 드래그 안 됨

**문제 사항**
- 업로드 후 첫 페이지만 렌더, 스크롤해도 빈 화면.
- 텍스트 선택이 안 돼서 드래그 번역 자체 불가능.

**해결 전략**
- pdf.js v4 API와 코드가 맞지 않아 텍스트 레이어가 예외를 던지고 페이지 루프가 끊겼다고 가정.
- 각 페이지 render 와 textLayer 호출을 try/catch 로 격리하고 v4 의 `TextLayer` 클래스로 전환.

**해결법**
- `renderOnePage(n)` 함수로 분리 + `pdfjsLib.TextLayer` 사용, devicePixelRatio 보정 추가.
- 키보드 네비 (↑↓←→ / Space / PageUp · Down / Home · End) 도 함께 와이어업.

## 3. 드래그 번역 응답이 5~7초로 느려서 즉시감 부족

**문제 사항**
- 모든 explain 호출이 채팅용 32B 모델로 가서 prompt eval + thinking 합쳐 5~7초.
- 단어 한 번 뽑는데 너무 비싸서 인터랙티브한 사전 경험이 안 됨.

**해결 전략**
- 작업별 모델 라우팅: 단어/문장 explain 은 작은 모델, 채팅은 사용자 선택 모델.
- 동시에 explain 요청의 `num_ctx` 도 4096으로 줄여서 첫 로드를 빠르게.

**해결법**
- `EXPLAIN_WORD_MODEL` / `EXPLAIN_SENTENCE_MODEL` env var 신설 (기본 `gemma3:4b`).
- warm 상태에서 약 1초 응답. 프론트는 explain 호출 때 chat 모델을 의도적으로 전송 안 함.

## 4. LLM 이 시스템 프롬프트를 무시하는 듯한 응답

**문제 사항**
- 한국어 지시·튜터 규칙을 시스템 프롬프트에 넣었는데 모델이 자기 식대로 영어·장문 답변.
- `/api/ps` 확인하니 모델이 `context_length=2048` 로 로드돼 있어 15K 짜리 시스템 프롬프트가 잘림.

**해결 전략**
- Ollama 서버의 글로벌 cap (`OLLAMA_CONTEXT_LENGTH`) 을 충분히 키우고 앱이 보내는 `num_ctx` 와 매칭.
- 디버그 UI 를 만들어서 “지금 어떤 프롬프트가 어느 컨텍스트로 갔는지” 한 번에 보이게.

**해결법**
- 컨테이너 env 의 `NUM_CTX` 를 모델 native max 에 맞춤(qwen3:32b → 40960, gemma4 → 32768).
- `/api/papers/<id>/debug` + `디버그` 모달이 토큰 합계 + 로드된 모델 컨텍스트를 표시.

## 5. 시스템 프롬프트가 거대 본문 뒤에 묻혀서 attention 희석

**문제 사항**
- 원래 구조는 `intro → <paper>{60KB}</paper> → 규칙` 이라 규칙이 가운데 본문 뒤로 밀림.
- Lost-in-the-middle 효과로 모델이 규칙을 거의 못 봄.

**해결 전략**
- 규칙·언어 지시를 프롬프트 **맨 앞** 으로 옮기고, 동일 규칙을 짧게 **맨 뒤** 에도 다시 박음.
- 본문은 "참고 자료" 라고 명시해서 지시문과 구별.

**해결법**
- `TUTOR_SYSTEM_PROMPT` 재구성: rules first → paper middle → reminder last.
- 응답이 한국어로 짧게, 마지막을 질문으로 끝맺는 비율이 크게 개선.

## 6. qwen3 thinking mode 가 응답을 지연시킴

**문제 사항**
- qwen3 / deepseek-r1 / gpt-oss 계열은 기본적으로 답하기 전에 길게 “think”.
- 응답 첫 토큰까지 10~40초가 비어 있어 행이 걸린 줄로 오해.

**해결 전략**
- Ollama 요청의 최상위 `think` 필드를 끌 수 있게 환경변수로 제어.

**해결법**
- `THINK_MODE` env var (`false` / `true` / `low` / `medium` / `high`), 기본 `false`.
- 필요 시(증명·복잡한 reasoning) 사용자가 `THINK_MODE=high` 로 켤 수 있게 보존.

## 7. PDF 업로드 직후 자동 개요 사라짐 (PoC 기능 회귀)

**문제 사항**
- 옛 코드는 업로드 직후 OVERVIEW 프롬프트로 자동 요약을 띄웠는데 리라이트하며 누락.
- 사용자가 매번 “개요 생성” 버튼을 다시 눌러야 해서 onboarding 흐름 깨짐.

**해결 전략**
- 자동 개요만 별도로 빠른 모델로 라우팅해서 첫 응답 지연을 줄임.
- 결과는 학습 로드맵까지 포함해 다음 액션을 유도.

**해결법**
- `OVERVIEW_MODEL` (기본 `gemma4:e4b`) + `fast_overview` 플래그 추가.
- 업로드 직후 자동 트리거, 채팅 영역에 5섹션 + 학습 로드맵 5–7항목 출력.

## 8. 글래스모피즘 팝업이 흰 PDF 위에서 묻힘

**문제 사항**
- 다크 테마용으로 만든 팝업 (dark glass) 이 흰 PDF 위에서 텍스트 거의 안 보임.
- 단순히 white-on-white 로 바꿨더니 이번엔 너무 밝아서 PDF 와 구분 안 됨.

**해결 전략**
- 약간 cool gray 톤(rgba(208, 215, 228, 0.62))으로 PDF 와 명도 차이를 주되 glass 느낌 유지.
- 내부 액션·결과 영역은 흰색 글래스로 두어 깊이감.

**해결법**
- `#sel-popup` / `#sel-action-popup` 공통 base + 부드러운 cool gray 배경.
- backdrop-filter blur(26px) saturate(180%) 로 PDF 텍스처를 살짝 흐림.

## 9. 외부 클릭·스크롤로 팝업이 의도치 않게 사라짐

**문제 사항**
- 팝업 외부 mousedown 또는 PDF 스크롤만 해도 자동으로 닫혀서 비교·확인이 어려움.
- DETAIL 영역에 결과를 띄워두고 다른 곳을 봐야 할 때 사라지는 게 가장 큰 불만.

**해결 전략**
- DETAIL 팝업은 명시적 닫기(X / Esc) 만으로 닫히게 변경.
- 기본 위치를 좌측 트리 옆 고정으로 두어 시야 방해 최소화.

**해결법**
- 외부 클릭·스크롤로 hide 하는 리스너 제거, Esc 핸들러 신설.
- `positionPopupDefault()` 가 트리 너비 기반으로 자동 위치 + 사용자가 드래그해서 옮길 수 있음.

## 10. 액션 버튼·결과를 한 팝업에 모아 산만함

**문제 사항**
- 드래그 직후 떠야 할 액션 버튼과, 결과를 오래 봐야 할 설명이 같은 팝업에 있어 정신없음.
- 새 드래그 때마다 결과 영역이 깜빡이고 위치도 자꾸 이동.

**해결 전략**
- 두 팝업으로 분리: MINI(드래그 시 선택 위치) = 액션만, DETAIL(좌측 상단) = 결과만.
- MINI 와 DETAIL 이 독립 상태·독립 닫기 동작을 갖도록.

**해결법**
- `#sel-action-popup` 신설, `handleActionClick(mode, selectionText, fromMini)` 공통 핸들러.
- MINI 는 외부 클릭으로 닫히지만 DETAIL 은 X/Esc 까지 유지, 단어장·노트 저장은 DETAIL 의 셀렉션 기준.

## 11. HTML/CSS 변경 후 새로고침해도 옛 화면이 그대로

**문제 사항**
- `templates/` 를 바인드 마운트했는데도 새 CSS 가 안 반영, 컨테이너 재시작해야만 보임.
- “HTML 수정은 새로고침이면 끝” 이라 말해놓고 실제론 안 됨.

**해결 전략**
- Flask 가 production 모드에서 Jinja 템플릿을 메모리 캐싱 (`debug=False` 일 때 default).
- `TEMPLATES_AUTO_RELOAD=True` 로 매 요청 mtime 체크해서 자동 재컴파일.

**해결법**
- `app.config["TEMPLATES_AUTO_RELOAD"] = True` 한 줄 추가.
- 이후 템플릿 수정은 컨테이너 재시작 없이 브라우저 새로고침만으로 반영.

## 12. qwen3:32b cold start 40초 — 인터랙티브 한계

**문제 사항**
- 채팅 첫 응답까지 모델 로드 30–40초 + prompt eval 10초가 누적돼 매번 답답.
- 64GB 메모리에 20GB 모델 + 32K KV cache 더라도 첫 토큰 지연이 너무 큼.

**해결 전략**
- 이 워크로드(논문 튜터·짧은 Q&A·드래그 사전) 에는 8B 가 충분하다고 판단하고 다운그레이드.
- 디스크도 20GB 회수, warm 상태 응답속도도 ~3배 개선.

**해결법**
- `ollama rm qwen3:32b` 후 기본 모델을 `gemma4:e4b` 로 교체, `NUM_CTX` 도 32768 로 정렬.
- 모델 드롭다운에서 embedding 모델 (`nomic-embed-text`) 은 필터링해 잘못 고르는 것 방지.

## 13. 패널 너비 고정 — PDF 폭이 좁고 채팅이 비좁음

**문제 사항**
- 익스플로러 260px / 채팅 380px 고정이라 긴 코드·표 보기 어려움.
- 사용자별 모니터·논문 크기에 맞춰 조정 불가.

**해결 전략**
- 좌·우 패널 경계를 드래그 핸들로 만들고 너비를 localStorage 에 보존.
- 더블클릭으로 기본값 리셋, 키보드는 안 잡고 마우스만 (간단).

**해결법**
- CSS 그리드를 `var(--left-w) 6px 1fr 6px var(--right-w)` 로, `.resizer` 요소 두 개 삽입.
- `setupResizer(handle, side)` 가 mousedown→move→up 트래킹 + min/max clamp + 저장.

## 14. PDF 줌 단축키·기본 배율 부재

**문제 사항**
- 줌은 툴바 버튼만 가능, 키보드 단축키 없음.
- 기본 배율 100% 가 본문 폰트 작아 가독성 떨어짐.

**해결 전략**
- 표준 PDF 뷰어 컨벤션 (Cmd/Ctrl + 휠) + 사용자가 요청한 (Cmd/Ctrl + 드래그) 둘 다 지원.
- 기본 배율을 120% 로 올려서 첫 인상 개선.

**해결법**
- 휠 줌은 디바운스, 드래그 줌은 mouseup 시점 한 번만 재렌더 (성능).
- `state.scale = 1.8` (= 120%) 기본값 + 페이지 캡션 초기 텍스트 동기화.

## 15. 단어 explain 이 “이 논문 안” 의미만 줘서 영어 학습용으로 약함

**문제 사항**
- 한 번 explain 받으면 논문 맥락 의미만 보고, 일반 사전적 의미·예문은 안 나옴.
- 영어 공부 겸용으로 쓰려는 사용자에겐 정보가 빠짐.

**해결 전략**
- explain 응답을 3 섹션 (📖 일반 뜻 / 📄 이 논문에서 / ✍️ 영어 예문) 으로 강제.
- 모르는 단어를 누적 저장하는 글로벌 단어장도 함께 추가.

**해결법**
- `EXPLAIN_WORD_PROMPT` 를 3 섹션 출력 포맷으로 재작성.
- `data/vocabulary.json` + `/api/vocabulary` CRUD + 상단바 `📚 단어장` 버튼 / 모달 (검색·펼침·삭제).

## 16. 한 번에 PDF 하나씩만 업로드 가능

**문제 사항**
- 폴더에 한 번에 여러 논문을 모아두기 위해 한 건씩 드래그해야 해서 번거로움.
- 다중 선택해도 한 파일만 처리되고 나머지는 무시됨.

**해결 전략**
- 백엔드 단건 업로드 엔드포인트는 유지, 프론트가 N 번 순차 호출.
- 다중 업로드 시 자동 개요는 스킵 — N 개를 자동 요약하면 비용·시간 모두 폭주.

**해결법**
- 파일 input 에 `multiple` 속성, 드래그 앤 드롭도 `dataTransfer.files` 전체 처리.
- `uploadFiles(files, folder)` 가 PDF 만 필터링하고 토스트로 진행 상황 표시.

## 17. 페이지 북마크 기능 부재

**문제 사항**
- 긴 논문에서 자주 다시 보는 페이지를 표시할 방법이 없음.
- “전에 봤던 그 figure 가 몇 페이지였더라” 매번 다시 탐색.

**해결 전략**
- 페이지 단위 북마크를 paper 디렉토리에 별도 JSON 으로 저장.
- 우측 패널에 `🔖 북마크` 탭 신설, 클릭으로 해당 페이지로 스크롤.

**해결법**
- `bookmarks.json` per paper + `/api/papers/<id>/bookmarks` CRUD.
- PDF 툴바 `🔖 추가` 버튼이 현재 보이는 페이지로 북마크 생성, 탭에서 라벨/페이지/삭제 관리.
